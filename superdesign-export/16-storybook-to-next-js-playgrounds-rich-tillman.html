Moving from Storybook to Internal Design Playgrounds.

Feb 14, 2024
15 min read
Rich Tillman

Why we decided to bring our component documentation directly into the monorepo using Next.js 14 App Router, and why Storybook might be overkill for your team.

For years, Storybook has been the industry standard for developing components in isolation. It’s powerful, feature-rich, and has a massive ecosystem. But as design systems mature, the friction of maintaining a separate build process, separate dependencies, and a disconnected documentation site becomes a burden.

The Multi-Repo Problem

In our previous setup at Airbnb, the design system lived in its own repository. Whenever a consumer application needed a change, they had to open a PR in the design system repo, wait for a release, and then update their `package.json`. This lifecycle is notoriously slow.

// The old way: Storybook stories in isolation
export default {
  title: 'Components/Button',
  component: Button,
} as Meta;

export const Primary = () => <Button variant="primary">Click me</Button>;

Enter Next.js Playgrounds

By moving to an internal Next.js-based playground within our Nx monorepo, we achieved three critical goals: Zero release latency, Shared environment consistency, and Full-stack documentation.

Now, a developer can modify a component and instantly see the impact in the playground using the exact same version of Tailwind, Next.js, and server-side logic that the main app uses.

// The new way: Next.js Documentation Routes
import { Button } from '@ui/core';
import { Playground } from '@/components/playground';

export default function Page() {
  return (
    <Playground 
      title="Primary Button"
      code={`<Button variant="primary">Click me</Button>`}
    >
      <Button variant="primary">Click me</Button>
    </Playground>
  );
}

The Technical Stack

We leveraged a few key technologies to make this transition seamless:

  • Shiki: For high-performance, beautiful code highlighting on the server.
  • Framer Motion: For orchestrating component previews and interaction states.
  • Next.js Parallel Routes: To display documentation side-by-side with live interaction tools.

The result? A 40% reduction in time-to-first-contribution for new engineers and a documentation site that is never out of sync with the codebase.
