The Challenge

Polyrepo chaos at scale

15 separate repositories, fragmented dependencies, 45-minute CI/CD builds, no shared patterns. Developers context-switched between repos daily, tooling was inconsistent, and shipping required cross-team coordination that stalled velocity.

Context-switching

Engineers juggled 5+ repos daily, losing focus and productivity with each switch.

Inconsistent tooling

Different ESLint configs, TS versions, and build setups across every repository.

No code sharing

Duplicate utilities across repos. Bug fixes required 15 separate PRs to propagate.

Release coordination

Cross-repo releases required manual coordination across teams. Releases took days.

The Solution

One monorepo. Unified velocity.

Migrated 15 fragmented repositories into a single Nx-powered monorepo with shared libraries, distributed caching, and automated publishing — fundamentally transforming how 50+ engineers build and ship software.

Phase 1

Migrated 15 repos

Consolidated all 15 repositories into a single Nx workspace with preserved git history and zero downtime.

Phase 2

Shared lib structure

Extracted 12 shared libraries (UI, utils, API, hooks) with clear boundaries and @company/* naming convention.

Phase 3

Incremental builds

Implemented Nx task graph with caching — only rebuilds what changed. 87% cache hit rate on average.

Phase 3

GitHub Actions + Nx Cloud

Distributed CI with Nx Cloud for remote caching. Parallel task execution across agent pool.

Phase 4

Semantic versioning

Automated version bumping and changelog generation via semantic-release. Zero manual versioning.

Phase 5

Onboarding playbook

Comprehensive docs, generators, and workshops. New engineers ship code on day one.

Decisions

Key decisions & trade-offs

Monorepo vs Polyrepo

Decision: Monorepo for 15-team org.

Rationale: Code sharing, unified tooling, and atomic commits outweighed the complexity. At 30+ engineers, polyrepo coordination costs exceed monorepo overhead.

Shared libs naming

Decision: @company/* scoped packages.

Rationale: Scoped naming prevents conflicts, signals ownership, and enables clean imports across all 8 apps. Convention over configuration.

CI strategy

Decision: Distributed builds via Nx Cloud.

Rationale: Central orchestration couldn't handle 8 apps + 12 libs. Distributed agents with remote caching cut build times by 60% and scaled with the team.

Versioning approach

Decision: Automated semantic versioning.

Rationale: Manual versioning across 12 libraries was error-prone and slow. semantic-release + conventional commits eliminated human error and generated changelogs automatically.
